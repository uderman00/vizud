Executive Overview – Contoso Sales Dashboard

This dashboard delivers a high‑level performance snapshot for Contoso’s leadership team. Key elements:

  • Revenue vs Target: a combined bar + line chart tracking monthly revenue against goal.
  • Product Structure Funnel: highlights how revenue cascades through product categories.
  • Store Revenue: two horizontal bar visuals separate brick‑and‑mortar stores from other sales channels.
  • Sales Channel Share: a pie chart showing each channel’s percentage contribution.
  • Revenue Heat Map: a choropleth map that pinpoints the top‑performing countries.
Built on a star schema and optimized DAX, the report supports full interactivity: slicers for date and geography, drill‑through to store‑level detail, and rich tooltips with margin and volume metrics. It demonstrates best practices for storytelling and performance with a data set exceeding two million rows.

Reducing Nested Iterators in DAX – A Hands‑On Example

Original query (non‑optimized)

EVALUATE
{
    SUMX(                    -- Outer iterator
        Store,
        SUMX(                -- Inner iterator
            Customer,
            [Sales Amount]   -- callback to FE
        )
    )
}

Optimized query

EVALUATE
{
    SUMX(
        SUMMARIZE (
            Sales,           -- single table
            Sales[CustomerKey],
            Sales[StoreKey]
        ),
        [Sales Amount]       -- computed in SE
    )
}
MetricOriginalOptimized
Total time (ms)45 26522 087
Formula Engine (FE)19 8413 760
Storage Engine (SE)25 42418 327
SE CPU129 90657 578
SE Parallelism× 5.1× 3.1

The first version uses a double iterator (SUMX(Store, SUMX(Customer…))). That forces the Formula Engine (FE) to loop over every possible combination of stores and customers, triggering expensive callback data caches to pull partial results from the Storage Engine (SE). Relationships between Sales, Store and Customer must also be materialized, adding extra cost.

By rewriting the query with SUMMARIZE(Sales, Sales[CustomerKey], Sales[StoreKey]) we:

  • Iterate only on existing pairs of Store × Customer – no wasted loops.
  • Stay entirely inside the Sales fact table, so relationship navigation is eliminated.
  • Let the SE calculate [Sales Amount], removing FE callbacks – FE just performs the final agregation.

The result is a 50 % reduction in total execution time and a FE workload that drops from 19.8 ms to 3.8 ms, demonstrating how query‑plan analysis and xmSQL inspection lead to practical, measurable gains.
